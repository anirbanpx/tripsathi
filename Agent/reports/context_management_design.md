### User Context Flow Analysis

Information enters the system at two points:
1. **Onboarding:** User answers 4–5 structured questions once — this becomes the persistent user profile
2. **Trip request:** User states trip parameters in natural language — this is per-session input

The profile from (1) must flow into every LLM call in (2). If persona context is lost between onboarding and planning, the Context Awareness Failure risk manifests immediately.

### Quality Risk Connection

**Priority risk:** Context Awareness Failure — the agent ignores stated soft constraints (kid age, elderly pace, budget) OR misses unstated local risks (Kamakhya queues, Shillong return timing).

This risk is entirely a **context management problem**:
- Angle A (explicit constraints ignored): user profile fields not passed correctly to planning calls
- Angle B (implicit local risks missed): RAG doesn't retrieve the right content, or retrieved content isn't surfaced in synthesis

Both angles are addressed by getting context schemas right.

### Discovered Context Patterns

- User profile is the critical shared context — must persist across all calls in Module 2
- RAG retrieval output is an intermediate context — generated by the system, consumed by synthesis
- Synthesis output is an intermediate context — generated by synthesis, consumed by plan generation
- Trip parameters are session-scoped — fresh per request, not persisted

## LLM Call Inventory

### LLM Call Analysis

**Call 1 — Persona Classification**
- **Core purpose:** Transform raw onboarding answers into structured user profile (persona type, autonomy mode, constraints)
- **Why LLM needed:** Users express constraints in natural language ("we have a toddler", "my parents are elderly") — inferring structured fields (kid_age=2, elderly=true, mobility_limited=true) requires language understanding, not if-else logic
- **Success criteria:** Persona type correctly classified; all hard constraints (kid age, mobility, dietary) extracted and structured; autonomy mode set appropriately
- **Failure modes:** Misses soft constraint from ambiguous phrasing; over-classifies vague answers; ignores implicit constraints ("we travel slow" → doesn't set low_pace=true)
- **Quality risk connection:** 8/10 — wrong persona = wrong context for every downstream call. Root cause of Context Awareness Failure.
- **Status:** ✅ Essential

**Call 2 — RAG Query Expansion**
- **Core purpose:** Expand trip request + user profile into specific retrieval queries for LlamaIndex
- **Why LLM needed:** Persona-aware query expansion ("Kerala trip" → "Kerala toddler-friendly activities", "Kerala family hotel in-house dining", "Kerala local food kids") is faster to build with LLM than robust templates for MVP
- **Success criteria:** Generated queries surface persona-relevant content from vector DB; queries include constraint-specific terms
- **Failure modes:** Generic queries miss persona context; over-generates queries causing noise
- **Quality risk connection:** 5/10 — affects what RAG retrieves but not the synthesis reasoning itself
- **Status:** 🔧 LLM for convenience — candidate to replace with deterministic templates post-MVP

**Call 3 — Research Synthesis**
- **Core purpose:** Synthesise RAG retrieved content + user profile into structured destination research (routing, key places, local risks, seasonal context, implicit warnings)
- **Why LLM needed:** Synthesises multiple retrieved documents, applies persona filtering, surfaces unstated local risks — requires reasoning across documents, not just extraction
- **Success criteria:** Routing logic explained with destination-specific reasoning; local risks surfaced proactively (Kamakhya queues, Shillong timing, operator fraud); persona constraints applied to filter inappropriate content
- **Failure modes:** Generic synthesis ignoring persona context; local risks not surfaced because RAG didn't retrieve relevant chunks; contradictory information from multiple sources not reconciled
- **Quality risk connection:** 9/10 — this is where Angle B of Context Awareness Failure passes or fails
- **Status:** ✅ Essential

**Call 4 — Plan Generation**
- **Core purpose:** Transform research synthesis + user profile + trip parameters into actionable structured plan (day-by-day itinerary, hotel shortlist with reasoning, budget breakdown, proactive warnings)
- **Why LLM needed:** Multi-step constraint satisfaction, sequencing, and personalised reasoning — no template handles simultaneous elderly + toddler constraints with destination-specific activity filtering
- **Success criteria:** Day plan adapts to all stated constraints; hotel recommendations include location reasoning; budget itemised; proactive warnings match real-trip friction points
- **Failure modes:** Ignores soft constraints in sequencing (same plan regardless of kid age); hotel reasoning doesn't update when group composition changes; budget not tracked across recommendations
- **Quality risk connection:** 10/10 — this IS the output. Context Awareness Failure manifests directly here.
- **Status:** ✅ Essential

### Calls to Remove or Defer

**Call 5 — Response Formatting** ❌ Removed
Call 4 should output structured JSON directly via LangGraph's structured output. Format in the React layer using templates — no additional LLM call needed. Each extra LLM call adds latency, cost, and failure surface.

### Call 4 Variants (added post-UX-review)

Call 4 (Plan Generation) has three distinct prompt variants — all share the same input
schema but differ in framing and additional context:

| Variant | When Called | Additional Context | Key Difference |
|---|---|---|---|
| Initial | Graph runs end-to-end on `/api/plan` | None beyond standard | Fresh plan from research |
| Refinement | `/api/refine` with feedback text | `previous_plan`, `refinement_history`, `user_feedback` | Apply specific change |
| Regenerate | `/api/regenerate` (no feedback) | `previous_plan` only | Notably different plan, anti-repetition prompt |

Plan output schema (all variants) extended with two fields:
- `hotels[].content_source: "rag" | "general"` — RAG-grounded vs base-model knowledge
- `hotels[].bookable: bool` + `days[].activities[].bookable: bool` — drives UI Book button visibility

## Context Waste Identification

### Issue 1 — Redundant field: `explicit_constraints` in Call 4

**Problem:** Call 4 receives both `user_profile` (which contains the full `constraints` object) AND a separately extracted `explicit_constraints` field that is a subset of `user_profile.constraints`. This passes the same data twice — the constraints are buried inside `user_profile` and also surfaced as a standalone field.

**Why it happened:** The intent was to keep hard constraints salient in the prompt so the LLM doesn't miss them inside a large profile object.

**Better approach:** Handle salience in the prompt template, not the schema. The Call 4 system prompt should explicitly instruct: "Extract and honour the following hard constraints from user_profile.constraints: kid_ages, elderly, mobility_limited, dietary, pace." This achieves the same salience without a redundant field.

**Resolution:** Remove `explicit_constraints` as a standalone input field in Call 4. Reference `user_profile.constraints` directly from the prompt template.

---

### Issue 2 — Type inconsistency: `trip_parameters` across Calls 2, 3, 4

**Problem:**
- Call 2: `{duration_nights, group_size, travel_dates}` — no `budget_total`
- Call 3: `{duration_nights, travel_dates, group_size}` — no `budget_total`
- Call 4: `{duration_nights, budget_total, travel_dates, group_size}` — includes `budget_total`

This means the same LangGraph state object is described with three different shapes across calls. Implementation risk: developer writes `trip_parameters` parsing once, but it won't match Call 4's expected shape.

**Resolution:** Standardize `trip_parameters` to the full object `{duration_nights: int, budget_total: int, travel_dates: string, group_size: int}` across all calls. Calls 2 and 3 simply won't reference `budget_total` in their prompts — having an unused field in context is cheaper than a type mismatch bug.

---

### Issue 3 — Retention policy: `research_synthesis` and `plan` overspecified for Sprint 2

**Problem:**
- `research_synthesis` is marked `session` retention. In Sprint 2 there's no plan-refinement flow (user can't ask "change day 3"). After Call 4 consumes it, it serves no purpose in state.
- `plan` (Call 4 output) is marked `session` retention. After FastAPI serialises it and sends it to React, the frontend holds the state. Keeping it in LangGraph state is wasteful.

**Resolution:** Both downgraded to `call_only` for Sprint 2. If Sprint 3 adds conversational plan refinement, revisit.

---

### Issue 4 — Optional field `expanded_queries` in Call 3 (YAGNI)

**Problem:** `expanded_queries` is passed to Call 3 as an optional diagnostic — "tells synthesis what was searched." In Sprint 2 there's no logging or gap analysis using this field, and synthesis doesn't structurally depend on it.

**Resolution:** Remove from Sprint 2 schema. Add back in Sprint 3 when observability tooling (LangSmith or similar) can actually use it.

---

### Issue 5 — `prior_trip_history` in Call 1 (dead field for Sprint 2)

**Problem:** `prior_trip_history` is defined as optional in Call 1's schema but has no data source in Sprint 2. It will never be populated. Including it creates a misleading schema.

**Resolution:** Remove from Sprint 2 schema. Reintroduce in Sprint 3 when persistent user profiles are implemented.
